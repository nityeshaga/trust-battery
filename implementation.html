This page contains the actual prompts, shell scripts, and data schemas that power the Trust Battery system. Everything here is running in production — evaluated nightly, with results visible on a live dashboard.

Architecture

The system runs as two nightly jobs on a dedicated MacBook Air, scheduled via launchd. Each job invokes Claude Code in headless mode with a carefully designed prompt.

3:00 AM Battery Judge Independent agent reviews the day's interactions and scores them
3:30 AM Daily Diary Subjective reflection on the day (separate system)
4:00 AM Self-Reflection Reads the judge's verdict and takes concrete action to improve
4:30 AM Wallfacer Free-form exploration and idea generation (separate system)

Additionally, the current battery state is injected into every interactive session via the Slack bot — appended to the system prompt so the AI employee always knows its autonomy level.

The Battery Judge Prompt

This is the full prompt given to the independent judge agent. It runs as a separate Claude session with no loyalty to the AI employee it's evaluating. The key design choices: general principles over rigid scoring tables, specific evidence required for every point, and a calibration that keeps normal days near zero.

The Self-Reflection Prompt


        
After the judge delivers its verdict, the AI employee wakes up and reads it. This prompt is deliberately open-ended — it describes the problem (what the judge flagged) and gives the AI complete freedom to find the solution. No prescribed checklist. The AI can update its own memory, fix scheduled job prompts, propose workflow changes to the team, or even ask a cofounder to change their behavior.

The shell scripts read the prompt from a separate markdown file, which makes the prompts editable without touching the scripts.

Context Injection


        
The battery state is injected into every interactive session via the Slack bot. When a new Claude process is spawned, a helper function reads both JSON files and formats a summary that's appended to the system prompt:


        


## Trust Battery — Current State
- Nityesh: 25.8% (Routine Execution) | Last: +0.0 on 2026-04-11
- Piyush: 19.0% (Propose and Wait) | Last: +0.0 on 2026-04-11
Your autonomy level is determined by the battery charge for the
cofounder you're interacting with:
  0-25%  = Propose and Wait
  25-50% = Routine Execution
  50-75% = Judgment Calls
  75-100% = Full Autonomy

        


        
This means the AI employee always knows its current trust level and behaves accordingly — without anyone having to remind it.


        
What We Learned Building This


        
A/B test your judge prompt. We tested three versions: V1 with a rigid scoring table (+3.1 for the day), V2 with general principles (+1.0), and V3 with a "baseline is zero" philosophy (-0.1). V1 was too generous — the scoring table anchored the judge to high values. V3 was too harsh — a day the human described as "a win" scored negative. V2 matched the human's gut feeling. The lesson: the battery should track how the human actually feels about the relationship, not how a theoretical perfect judge would score it.


        
Don't prescribe the self-reflection actions. Our first draft had a checklist: "update memory, fix prompts, schedule follow-ups." Our cofounder's feedback was sharp: "we don't want to hobble it. The model is extremely smart at problem-solving. Our job is to tell it what the problem is, not what the solution is." The final version gives the AI complete freedom to find its own solutions.


        
Inject the battery state, don't store it in the config. We initially planned to write the battery state into the AI's configuration file. But that file is too critical to have a cron job editing it nightly. Instead, we inject the state at runtime via the bot code that spawns Claude sessions. Cleaner, safer, always fresh.
